1. MVVM-Architektur Umsetzung​

Die App folgt dem MVVM-Pattern mit drei klar getrennten Schichten. Models definieren die Datenstrukturen, ViewModels enthalten die Business-Logik, und Views sind nur für die Darstellung zuständig.

Model Layer (Datenmodelle)​

Die Models sind reine TypeScript-Interfaces ohne Logik - sie definieren nur die Form der Daten:

// src/models/index.ts

export interface ViewModelBase {
  subscribe(callback: () => void): () => void;
  notifyChange(): void;
}

export interface PurchaseItem {
  id: string;
  name: string;
  price: number;
  description: string;
  category: string;
}

export interface GameState {
  currentBudget: number;
  purchasedItems: PurchaseItem[];
  totalSpent: number;
  gameCompleted: boolean;
  startTime: Date;
  endTime?: Date;
}

export interface Statistics {
  gamesPlayed: number;
  fastestCompletion?: number;
  averageItemsPerGame: number;
  favoriteCategory?: string;
  totalMoneySpent: number;
  mostExpensiveItem?: PurchaseItem;
  totalItemsPurchased: number;
}

ViewModel Layer (Business Logic)​

BaseViewModel:

Alle ViewModels erben von dieser Basis-Klasse, die das Observer Pattern implementiert. Views können sich subscriben und werden automatisch über Änderungen informiert:

// src/viewmodels/BaseViewModel.ts

export abstract class BaseViewModel implements ViewModelBase {
  private subscribers: (() => void)[] = [];
  
  subscribe(callback: () => void): () => void {
    this.subscribers.push(callback);
    
    return () => {
      const index = this.subscribers.indexOf(callback);
      if (index > -1) {
        this.subscribers.splice(index, 1);
      }
    };
  }

  notifyChange(): void {
    this.subscribers.forEach(callback => callback());
  }

  dispose(): void {
    this.subscribers = [];
  }
}

GameViewModel:

Das GameViewModel enthält die gesamte Spiellogik. Hier ein Beispiel für die purchaseItem-Methode:

// src/viewmodels/GameViewModel.ts

export class GameViewModel extends BaseViewModel {
  private gameState: GameState;
  private statistics: Statistics;

  getGameState(): GameState {
    return { ...this.gameState };  // Defensive Copy
  }

  purchaseItem(item: PurchaseItem): boolean {
    if (this.gameState.currentBudget >= item.price) {
      // Budget abziehen
      this.gameState.currentBudget -= item.price;
      this.gameState.totalSpent += item.price;
      this.gameState.purchasedItems.push({ ...item });

      // Statistiken aktualisieren
      this.updateStatisticsOnPurchase(item);

      // Spiel abgeschlossen?
      if (this.gameState.currentBudget === 0) {
        this.gameState.gameCompleted = true;
        this.gameState.endTime = new Date();
        this.updateStatisticsOnCompletion();
      }

      // Speichern und Views benachrichtigen
      this.saveGameState();
      this.notifyChange();
      return true;
    }
    return false;
  }
}

Die Methode prüft zuerst ob genug Budget vorhanden ist, aktualisiert dann den State, speichert automatisch und benachrichtigt alle Views über die Änderung.

View Layer (UI Components)​

Views erhalten ViewModels als Props und subscriben sich zu deren Änderungen:

// src/views/MainGameView.tsx

interface MainGameViewProps {

export function MainGameView({ viewModel }: MainGameViewProps) {

        )}
      />

  );
}

2. Navigation-Implementierung​

Die App nutzt Expo Router mit file-based navigation - die Ordnerstruktur definiert automatisch die Routes. Das ist wartbarer und typsicherer als programmatisches Routing.

Navigations-Architektur​

Die App hat eine hierarchische Navigationsstruktur:

Root Layout (Stack Navigator)
    ├── (tabs) - Tab Navigator [HIDDEN HEADER]
    │   ├── index.tsx (Game Screen)
    │   ├── statistics.tsx (Statistics Screen)
    │   └── settings.tsx (Settings Screen)
    └── modal.tsx - Modal Screen

  • Tab Navigator [HIDDEN HEADER]
  • index.tsx (Game Screen)
  • statistics.tsx (Statistics Screen)
  • settings.tsx (Settings Screen)
  • modal.tsx - Modal Screen

Warum dieser Aufbau?

  • Stack Navigator als Root ermöglicht Modals über den Tabs
  • (tabs) Route-Gruppe versteckt den Namen aus der URL
  • Tab Navigator ermöglicht schnellen Wechsel zwischen Hauptscreens
  • Shared ViewModels über Context = kein Props-Drilling

Root Layout (Stack Navigator)​

Der Root-Layout definiert den grundlegenden Navigations-Container und das Theming:

// app/_layout.tsx

import { DarkTheme, DefaultTheme, ThemeProvider } from '@react-navigation/native';
import { Stack } from 'expo-router';
import { StatusBar } from 'expo-status-bar';
import { useColorScheme } from '@/hooks/use-color-scheme';

export const unstable_settings = {
  anchor: '(tabs)',  // Startpunkt der App
};

export default function RootLayout() {
  const colorScheme = useColorScheme();

  return (
    <ThemeProvider value={colorScheme === 'dark' ? DarkTheme : DefaultTheme}>
      <Stack>
        {/* Tab Navigator als Hauptnavigation */}
        <Stack.Screen name="(tabs)" options={{ headerShown: false }} />
        
        {/* Modal Screen für zukünftige Features */}
        <Stack.Screen 
          name="modal" 
          options={{ 
            presentation: 'modal', 
            title: 'Modal' 
          }} 
        />
      </Stack>
      <StatusBar style="auto" />
    </ThemeProvider>
  );
}

Key Features:

  • Stack Navigator: Ermöglicht Modal-Screens über den Tabs
  • headerShown: false: Tabs brauchen keinen Header
  • unstable_settings.anchor: Definiert die Start-Route

Tab Navigation mit Shared ViewModels​

Die Tab-Navigation ist der Core der App. Hier werden die ViewModels zentral erstellt und über Context geteilt:

// app/(tabs)/_layout.tsx

import MaterialIcons from '@expo/vector-icons/MaterialIcons';
import { Tabs } from 'expo-router';
import React, { createContext, useContext, useState } from 'react';
import { GestureHandlerRootView } from 'react-native-gesture-handler';

import { HapticTab } from '@/components/haptic-tab';
import { Colors } from '@/constants/theme';
import { useColorScheme } from '@/hooks/use-color-scheme';
import { GameViewModel, SettingsViewModel, StatsViewModel } from '@/src/viewmodels';

// Context für Shared ViewModels
const ViewModelsContext = createContext<{
  gameViewModel: GameViewModel;
  settingsViewModel: SettingsViewModel;
  statsViewModel: StatsViewModel;
} | null>(null);

// Custom Hook für typsicheren Zugriff
export const useViewModels = () => {
  const context = useContext(ViewModelsContext);
  if (!context) {
    throw new Error('useViewModels must be used within a ViewModelsProvider');
  }
  return context;
};

export default function TabLayout() {
  const colorScheme = useColorScheme();

  // ViewModels werden EINMAL erstellt und geteilt
  const [gameViewModel] = useState(() => new GameViewModel());
  const [settingsViewModel] = useState(() => new SettingsViewModel());
  const [statsViewModel] = useState(() => new StatsViewModel());

  const viewModels = {
    gameViewModel,
    settingsViewModel,
    statsViewModel,
  };

  return (
    <ViewModelsContext.Provider value={viewModels}>
      <GestureHandlerRootView style={{ flex: 1 }}>
        <Tabs
          screenOptions={{
            tabBarActiveTintColor: Colors[colorScheme ?? 'light'].tint,
            headerShown: false,
            tabBarButton: HapticTab,  // Custom Button mit Haptic Feedback
          }}>
          <Tabs.Screen
            name="index"
            options={{
              title: 'Game',
              tabBarIcon: ({ color }) => (
                <MaterialIcons size={28} name="gamepad" color={color} />
              ),
            }}
          />
          <Tabs.Screen
            name="statistics"
            options={{
              title: 'Statistics',
              tabBarIcon: ({ color }) => (
                <MaterialIcons size={28} name="bar-chart" color={color} />
              ),
            }}
          />
          <Tabs.Screen
            name="settings"
            options={{
              title: 'Settings',
              tabBarIcon: ({ color }) => (
                <MaterialIcons size={28} name="settings" color={color} />
              ),
            }}
          />
        </Tabs>
      </GestureHandlerRootView>
    </ViewModelsContext.Provider>
  );
}

Warum dieser Ansatz?

  1. Single Source of Truth: ViewModels werden nur einmal erstellt
  2. Shared State: Alle Tabs arbeiten mit den gleichen Daten
  3. Automatic Updates: Änderungen in einem Tab sind sofort in allen sichtbar
  4. Type Safety: TypeScript prüft die Context-Nutzung zur Compile-Zeit
  5. Kein Props-Drilling: Direkte Injection über Custom Hook

Beispiel-Flow:

  1. User kauft Item im Game Tab
  2. GameViewModel.purchaseItem() wird aufgerufen
  3. GameState wird aktualisiert
  4. notifyChange() benachrichtigt alle Subscriber
  5. Statistics Tab zeigt neue Werte (wenn aktiv)

export function HapticTab(props: BottomTabBarButtonProps) {

Shake to Undo​

Funktion im Vieo gezeigt. Danach bekommt man diesen Bildschirm.

Gerät schütteln um den letzten Kauf rückgängig zu machen.

// src/views/MainGameView.tsx

useEffect(() => {
  let lastShakeTime = 0;
  const SHAKE_THRESHOLD = 1.5;
  const SHAKE_COOLDOWN = 1000;

  Accelerometer.setUpdateInterval(100);
  
  const subscription = Accelerometer.addListener(({ x, y, z }) => {
    // Magnitude berechnen
    const acceleration = Math.sqrt(x * x + y * y + z * z);
    const now = Date.now();
    
    if (acceleration > SHAKE_THRESHOLD && now - lastShakeTime > SHAKE_COOLDOWN) {
      lastShakeTime = now;
      handleShake();
    }
  });

  return () => {
    subscription.remove();
  };
}, [gameState.purchasedItems]);

const handleShake = async () => {
  if (gameState.purchasedItems.length > 0) {
    const lastItem = gameState.purchasedItems[gameState.purchasedItems.length - 1];
    const success = viewModel.sellItem(lastItem);
    
    if (success) {
      await Haptics.notificationAsync(Haptics.NotificationFeedbackType.Success);
      Vibration.vibrate(50);
      showStyledAlert(
        'Undo Successful', 
        `Returned: ${lastItem.name} (+$${lastItem.price.toLocaleString()})`
      );
    }
  }
};

Wie funktioniert Shake-Detection?

  1. Accelerometer liefert x, y, z Werte (in G)
  2. Ich berechne die Magnitude: √(x² + y² + z²)
  3. Wenn Magnitude > 1.5G -> Shake erkannt
  4. Cooldown verhindert Doppel Trigger
  5. Letztes Item wird zurückverkauft

Normal liegt das Gerät bei etwa 1G (Erdanziehung). Beim Schütteln steigt die Beschleunigung deutlich an.

Der Beschleunigungssensor misst die Bewegung des Geräts in drei Achsen:

import { Accelerometer } from 'expo-sensors';

  // x: Links/Rechts
  // y: Oben/Unten  
  // z: Vorne/Hinten
  
  const acceleration = Math.sqrt(x * x + y * y + z * z);
  
  if (acceleration > SHAKE_THRESHOLD) {
    // Shake!
  }
});

Beispiel-Werte:

  • Normal: (0, 0, 1) -> 1G nach unten
  • Shake: x=0.8, y=1.2, z=0.9 -> √(0.64+1.44+0.81) = 1.89 -> Shake erkannt!

5. Persistente Datenspeicherung​

AsyncStorage speichert die Daten lokal auf dem Gerät als JSON.

Speichern​

// src/viewmodels/GameViewModel.ts

async saveGameState(): Promise<void> {
  try {
    // Dates zu ISO Strings konvertieren
    const gameStateToSave = {
      ...this.gameState,
      startTime: this.gameState.startTime.toISOString(),
      endTime: this.gameState.endTime?.toISOString(),
    };
    
    // Als JSON speichern
    const jsonValue = JSON.stringify(gameStateToSave);
    await AsyncStorage.setItem('@gameState', jsonValue);
    
    console.log('Game state saved successfully');
  } catch (e) {
    console.error('Failed to save game state', e);
  }
}

Laden​

async loadGameState(): Promise<void> {
  try {
    const jsonValue = await AsyncStorage.getItem('@gameState');
    
    if (jsonValue != null) {
      const loadedState = JSON.parse(jsonValue);
      
      // ISO Strings zurück zu Dates konvertieren
      this.gameState = {
        ...loadedState,
        startTime: new Date(loadedState.startTime),
        endTime: loadedState.endTime ? new Date(loadedState.endTime) : undefined,
      };
      
      console.log('Game state loaded successfully');
    }
  } catch (e) {
    console.error('Failed to load game state', e);
  }
}

Kritische Beurteilung​

The good​

Die MVVM-Architektur funktioniert wirklich gut. Views und Logik sind sauber getrennt, das Observer Pattern sorgt für automatische Updates, und durch BaseViewModel muss ich den Subscribe-Mechanismus nur einmal implementieren. Die Navigation mit Expo Router ist elegant - die Ordnerstruktur definiert automatisch die Routes, und durch den Context können alle Tabs auf die gleichen ViewModels zugreifen ohne Props durch mehrere Ebenen durchzureichen.

Die Shake-to-Undo Funktion ist das Highlight. Der Accelerometer-Algorithmus mit 1.5G Threshold und 1-Sekunden Cooldown funktioniert zuverlässig und fühlt sich natürlich an. Das haptische Feedback macht einen riesigen Unterschied - die verschiedenen Patterns (Light, Medium, Success, Error) sind gut unterscheidbar und machen die App "fühlbar".

Die Persistenz mit AsyncStorage ist simpel aber robust. Auto-Save nach jeder Aktion bedeutet kein Datenverlust, und die JSON-Struktur lässt sich leicht erweitern. Das Offline-First Design ist perfekt für eine Mobile App.

The bad​

Bei der MVVM-Architektur habe ich keine Dependency Injection implementiert - ViewModels werden einfach mit new erstellt. Das macht Unit-Testing schwieriger, weil ich die Dependencies nicht mocken kann. In einem grösseren Projekt würde ich ein DI-Framework nutzen.

Die Gestiksteuerung ist relativ basic. Ich habe nur Tap, Shake und Scroll - Swipe-to-Delete, Long-Press oder Pinch-to-Zoom fehlen. Das wäre cool zu haben, war aber nicht im Scope. Der Accelerometer läuft nur wenn die MainGameView aktiv ist (durch useEffect cleanup), aber mit 10 Hz Update-Rate kann das trotzdem die Batterie belasten. In einer Production-App würde ich die Update-Rate reduzieren wenn keine Interaktion stattfindet.

Die Kamera-Funktion ist sehr simpel - nur Foto aufnehmen, keine Filter, kein Zoom, keine Galerie-Integration. Für ein Profilbild reicht es, aber mehr Features wären nice.

Die Persistenz ist rein lokal ohne Cloud-Sync. Wenn jemand die App deinstalliert oder das Gerät wechselt, sind alle Daten weg. AsyncStorage hat auch keine Schema-Versionierung - wenn sich die Datenstruktur ändert, müsste ich manuell migrieren. Für sehr viele Items wäre SQLite performanter.

Fazit​

Die Implementierung erfüllt alle Anforderungen von HZ3:

  • MVVM-Architektur sauber implementiert mit Observer Pattern
  • Navigation mit Expo Router und Shared Context funktioniert einwandfrei
  • Gestiksteuerung (Touch + Shake) ist intuitiv und funktional
  • 2 Sensoren: Accelerometer (Shake) + Camera (Profilbild)
  • 2 Aktoren: Haptic Engine + Vibration mit differenzierten Patterns
  • Persistenz mit AsyncStorage und Auto-Save ist zuverlässig
